2. 模块责任边界

输入

SnipeImportBatch、Snipe-IT API configuration、SnipeImportOptions。

输出

SnipeImportResult，包含 created/updated/skipped/failed 计数、actions、failures、warnings。

核心职责

确保 company、asset model、hardware asset 存在；根据比较结果创建或更新 asset。

明确不属于本模块

  • 不从 Atera 拉数据。
  • 不执行 Atera 到 Snipe-IT 的字段 mapping。
  • 不决定 sync schedule。
  • 不写 status.json 或其它状态存储。
  • 不负责发送 email 或 UI 展示。

4. Snipe-IT 比较逻辑

“比较”指把一个 SnipeAssetImportRecord 与 Snipe-IT 现有数据进行查找和决策。 当前总策划案定义的资产身份优先级是：先按 serial 查找，找不到时再按 generated asset tag 查找。

资产查找顺序：Serial 存在时优先查 serial；未命中或 serial 不存在时，使用 AssetTag 作为 fallback identity。
比较对象 查找键 命中时 未命中时 注意事项
Company CompanyName 复用已存在 company。 若 CreateMissingCompanies 为 true，则计划创建；否则该资产应跳过或失败。 实际 search/create endpoint、唯一性规则、返回 body 必须查官方文档。
Asset Model ManufacturerName + ModelName + CategoryName 复用已存在 model。 若 CreateMissingModels 为 true，则计划创建；否则该资产应跳过或失败。 manufacturer/category/model 的依赖关系需要在技术规格中写清。
Hardware Asset 优先 Serial，fallback AssetTag 更新 asset。 创建新的 hardware asset。 如果 serial 和 asset tag 命中不同 asset，技术规格必须定义冲突处理策略。

5. 导入流程

1 接收 Batch 验证 batch、options 和 cancellation token。
2 逐条处理 Asset 读取 Reconstruction 已生成的 SnipeAssetImportRecord。
3 Ensure Company 查找 company；按 option 决定是否创建缺失 company。
4 Ensure Model 查找 manufacturer/model/category 组合；按 option 决定是否创建缺失 model。
5 Compare Asset 按 serial，再按 asset tag fallback 查找硬件资产。
6 Create / Update dry-run 只记录动作；非 dry-run 执行写入并解析响应。

每条资产的建议处理顺序

  1. 如果 DryRun 为 true，仍执行必要的只读比较，以产生 planned actions；不执行 create/update 写入。
  2. 确认 company。缺失且不允许创建时，不继续 asset create/update，返回 skipped 或 failure，具体分类需在技术规格中固定。
  3. 确认 asset model。缺失且不允许创建时，不继续 asset create/update。
  4. 按 serial 查找 asset；如果 serial 缺失或未命中，再按 asset tag 查找。
  5. 命中现有 asset 时更新；两个身份命中不同资产时不要自动选择，建议记录失败。
  6. 未命中任何 asset 时创建。
  7. 读取 Snipe-IT JSON response body；即使 HTTP status 是 200，也要识别 body 中表示失败的 payload。
  8. 把每个动作、失败和 warning 汇总到 SnipeImportResult。

6. Dry-Run 行为

阶段 Dry-run 应做 Dry-run 不应做
比较阶段 可以执行只读查询，计算将会创建/更新/跳过哪些目标。 不应修改 Snipe-IT 中任何 company、model 或 asset。
依赖对象 记录 planned company/model create actions，WasExecuted = false。 不应真的创建缺失 company/model。
资产 记录 planned asset create/update actions，保留 target name 和说明。 不应发送任何 create/update 请求。
结果 DryRun = true；actions 表示计划动作；计数语义需要在技术规格中固定为 planned counts 或 executed counts。 不应让调用者误以为写入已经执行。

7. 错误与结果聚合

  • HTTP request 失败、取消、超时、认证失败、response body 无法解析，都应转换为结构化 failure 或抛出规格定义的异常。
  • 官方文档确认前，不应假设 Snipe-IT 的错误字段名、分页字段名或成功/失败 body 形状。
  • ImportFailure 至少记录 TargetType、TargetName、Message、可选 Code。
  • ImportAction 应记录动作类型、目标类型、目标名称、是否执行、可选说明。
  • batch 自带 warnings 应并入最终 SnipeImportResult.Warnings，Import 模块新增 warning 也要保留来源。
