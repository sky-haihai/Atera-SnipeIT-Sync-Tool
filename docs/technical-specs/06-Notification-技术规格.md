# Notification - 技术规格

## 2026-07 SyncRunResult projection

`NotificationRequestFactory.BuildMessage` must set `DryRun` from `SyncRunResult.DryRun` and `FailedAssets` from `SyncRunResult.ImportResult?.FailedAssets ?? 0`. Run-level failures remain in `FirstFailure` and are never converted into asset failures. Required regression coverage includes a Mapping failure with zero failed assets and a dry-run value that remains identical across notification, history and IPC projections.

## 1. Scope

Notification Module 为 sync runtime 提供通知请求构造、事件过滤和可替换 publisher 边界。

本技术规格基于：

- `docs/module-plans/06-Notification-功能职责.md`
- `docs/AteraSnipeSync_AI_Agent_Master_Plan.md` 的 `Module 6 - Notification Module`
- existing `src/AteraSnipeSync.Core/Notifications/NotificationRequest.cs`
- existing `src/AteraSnipeSync.Core/Notifications/Interfaces/INotificationPublisher.cs`
- existing `src/AteraSnipeSync.Core/Configuration/NotificationConfig.cs`

第一版仅提供 stub-safe pipeline；自第 15–16 节 concrete delivery revision 起，真实 SMTP email、Teams Workflow Adaptive Card 与 generic HTTPS webhook 是当前生产 contract，第 8 节 Null publisher 仅保留为测试/显式禁用替身。

本模块不新增或修改 Atera/Snipe-IT API wire contracts，不新增真实 API 测试。

Property rules:

- `EventType` must be a non-blank stable event string from `NotificationEventTypes` when generated by this module
- `Severity` must be `Information`, `Warning`, `Error`, or `Critical`
- `Subject` must be non-blank and short enough for email/webhook titles
- `Message` must be non-blank and safe to log
- `SyncResult` may be `null` for future non-sync notifications

- `OnEvents` must not be `null`

## 6. Request Factory

### 6.1 `NotificationRequestFactory`

File:

```text
src/AteraSnipeSync.Core/Notifications/NotificationRequestFactory.cs
```

Signature:

```csharp
public static class NotificationRequestFactory
{
    public static NotificationRequest CreateForSyncResult(
        SyncRunResult result,
        string triggeredBy);
}
```

Required comment:

```csharp
/// <summary>
/// Builds safe, user-readable notification requests from completed sync run results.
/// </summary>
```

Validation:

- throw `ArgumentNullException` when `result` is `null`
- throw `ArgumentException` when `triggeredBy` is null/empty/whitespace

Event mapping:

```text
triggeredBy = scheduled, result.Success = true  -> ScheduledSyncCompleted
triggeredBy = scheduled, result.Success = false -> ScheduledSyncFailed
triggeredBy = manual, result.Success = true     -> ManualSyncCompleted
triggeredBy = manual, result.Success = false    -> ManualSyncFailed
triggeredBy = manual-preview, success           -> ManualPreviewCompleted
triggeredBy = manual-preview, failed            -> ManualPreviewFailed
unknown trigger, success                        -> SyncCompleted
unknown trigger, failed                         -> SyncFailed
```

`triggeredBy` comparison must trim and use ordinal ignore-case.

Severity mapping:

- `Information` when `result.Success = true`, `result.Warnings.Count = 0`, and `result.Failures.Count = 0`
- `Warning` when `result.Success = true` and warnings or structured record failures are present
- `Error` when `result.Success = false`
- `Critical` when `result.Success = false` and any `SyncFailure.Code` clearly identifies authentication, authorization, or credential failure

First implementation may treat these failure codes as critical:

- `AuthenticationFailed`
- `Unauthorized`
- `Forbidden`
- `InvalidApiKey`
- `InvalidToken`

Subject mapping:

- scheduled success: `Scheduled sync completed`
- scheduled failure: `Scheduled sync failed`
- manual success: `Manual sync completed`
- manual failure: `Manual sync failed`
- manual preview success: `Manual preview completed`
- manual preview failure: `Manual preview failed`
- generic success: `Sync completed`
- generic failure: `Sync failed`

Message must include:

- event result text
- `StartedAt` in UTC round-trip format
- `FinishedAt` in UTC round-trip format
- pulled count: `result.PullResult?.Summary.AgentCount ?? 0`
- mapped count: `result.ImportBatch?.Summary.MappedAssetCount ?? 0`
- created assets: `result.ImportResult?.CreatedAssets ?? 0`
- updated assets: `result.ImportResult?.UpdatedAssets ?? 0`
- deleted assets: `result.ImportResult?.DeletedAssets ?? 0`
- skipped assets: `result.ImportResult?.SkippedAssets ?? 0`
- failed assets: `result.ImportResult?.FailedAssets ?? 0`
- warning count: `result.Warnings.Count`
- first failure stage/code/message when failures exist

Message must not include:

- API keys
- tokens
- authorization headers
- raw Atera JSON
- raw Snipe-IT JSON
- full asset dumps
- full local configuration JSON

```csharp
/// </summary>
```

Constructor responsibilities:

- validate `logger` is not null

`PublishAsync` responsibilities:

- validate `request` is not null
- validate `EventType`, `Severity`, `Subject`, and `Message` are non-blank
- call `cancellationToken.ThrowIfCancellationRequested()` before logging
- write a debug log with event type and severity
- do not call HTTP, SMTP, file system, Atera API, Snipe-IT API, Teams, Slack, or any webhook
- return `Task.CompletedTask`

Logs must not include raw `Message` if future messages may contain user-entered text. First implementation may log only event type, severity, and subject.

## 9. Integration Rules

Worker Scheduler and future manual sync flows may use:

```text
SyncRunResult
  -> NotificationRequestFactory.CreateForSyncResult(...)
  -> NotificationEventFilter.ShouldPublish(...)
  -> INotificationPublisher.PublishAsync(...)
```

Notification publish should happen after status save when a completed run result exists.

If notification publishing fails:

- log the notification failure
- do not rewrite the original `SyncRunResult`
- let the caller decide whether the runtime loop continues

Cancellation should stop publishing immediately and propagate `OperationCanceledException`.

## 10. Logging

Use `ILogger<NullNotificationPublisher>` for null publisher logs.

Required log events:

- Debug: null publisher accepted a notification request
- Error: future concrete senders should log failed send attempts before throwing

Logs must not include secrets, auth headers, raw payloads, full config content, or full asset inventories.

## 15. 2026-07 Concrete Email/Webhook delivery

### 15.1 Configuration contract

`NotificationConfig` adds optional `SmtpHost`, `SmtpUsername`, `SmtpPassword`, `EmailFrom`, existing `EmailTo` and `WebhookUrl`, plus `SmtpPort = 587` and `SmtpUseSsl = true` defaults。Email is configured when any email field is present and then requires host、valid port、from、at least one recipient；username/password are an all-or-none pair。Webhook is configured only by a non-blank absolute HTTPS URL。

### 15.2 Channel boundary

```csharp
public interface INotificationChannelSender
{
    string ChannelName { get; }
    bool IsConfigured(NotificationConfig config);
    Task SendAsync(
        NotificationConfig config,
        NotificationRequest request,
        CancellationToken cancellationToken);
}

public interface INotificationTester
{
    Task<NotificationTestResult> TestAsync(
        NotificationConfig config,
        CancellationToken cancellationToken);
}
```

`EmailNotificationSender` validates addresses and creates one plain-text envelope。`SystemNetSmtpNotificationTransport` owns `SmtpClient`/`MailMessage` disposal and performs exactly one send。An injected `ISmtpNotificationTransport` allows offline envelope assertions。

`WebhookNotificationSender(HttpClient)` posts one JSON object with properties `eventType`, `severity`, `subject`, `message`, numeric `deleted`, `occurredAtUtc` and content type `application/json`。It uses response headers completion, treats only 2xx as success, does not read/return response bodies, and never puts the URL in result/log messages。

The Worker named Notifications `HttpClient` removes default request loggers so a secret-bearing webhook URL is not emitted by framework HTTP logs。OAuth2/Graph email、custom webhook auth/signature、authenticated Teams trigger/run-history access、Slack payload adaptation and automatic retry remain out of scope。

`CompositeNotificationPublisher` implements `INotificationPublisher` and `INotificationTester`：normal `PublishAsync(request, config, token)` attempts every configured channel, continues after channel failure, and throws one sanitized aggregate delivery exception after all attempts；zero configured channels is configuration failure。`TestAsync` builds a fixed safe test request, independently attempts Email/Webhook, and returns two `NotificationChannelTestResult` objects with `Configured`, `Succeeded`, and bounded sanitized `Message`。

### 15.3 Runtime integration

Worker DI replaces `NullNotificationPublisher` with one `CompositeNotificationPublisher` instance registered as both `INotificationPublisher` and `INotificationTester`。`WorkerScheduler` passes its current runtime `NotificationConfig` snapshot into `PublishAsync`；legacy `SyncScheduler` passes `_notificationConfig`。

IPC adds payload-free `TestNotifications` and terminal `NotificationTest`。Worker loads `NotificationConfig` from local JSON, shares the global run coordinator under `notification-test`, and returns only channel status；SMTP credentials, recipients and webhook URL never enter IPC。Tray first validates/saves the complete current form, sends `ReloadSchedule`, then starts `TestNotifications`。

### 15.4 Required tests

- email validation and one fake-transport envelope with TLS/auth/from/recipient/subject/body。
- webhook JSON/content-type/method success, non-2xx failure, HTTPS validation and no body leakage using fake handler。
- composite both-channel attempt, partial failure continuation, no-channel failure, cancellation and per-channel test results。
- local settings round-trip for every SMTP/email/webhook property without printing secrets。
- Worker command payload-free terminal result, busy/cancel behavior and no dependency on Atera/Snipe settings。
- Tray/IPC result validation and UI manual acceptance；automated tests never contact real SMTP/webhook。

## 16. 2026-07 Teams Workflow Adaptive Card payload

Add `AteraSnipeSync.Core.Configuration.WebhookPayloadFormat` with values `TeamsAdaptiveCard` and `GenericJson`。`NotificationConfig.WebhookPayloadFormat` defaults to `TeamsAdaptiveCard`。`LocalAppSettingsStore` writes the enum name as `Notifications.WebhookPayloadFormat`; a missing value loads as `TeamsAdaptiveCard`, while an unknown value is rejected as invalid local configuration instead of silently selecting a wire format。

`WebhookNotificationSender.SendAsync` continues to validate one absolute HTTPS endpoint and issue exactly one POST with `application/json`。It selects the body by `WebhookPayloadFormat`:

- `TeamsAdaptiveCard`: `{ "type": "message", "attachments": [{ "contentType": "application/vnd.microsoft.card.adaptive", "contentUrl": null, "content": { "$schema": "http://adaptivecards.io/schemas/adaptive-card.json", "type": "AdaptiveCard", "version": "1.2", "body": [...] } }] }`。The body contains a wrapped subject TextBlock, a FactSet for severity/event/UTC timestamp, and a wrapped message TextBlock。
- `GenericJson`: use `{ eventType, severity, subject, message, deleted, occurredAtUtc }`, where `deleted` is numeric。

No sender response model claims downstream delivery。A 2xx response completes `SendAsync`; non-2xx remains a sanitized `NotificationDeliveryException` containing status only。`CompositeNotificationPublisher.TestAsync` returns a bounded message saying the channel endpoint/server accepted the request。The Tray renders configured success as `Accepted`, because a Teams Workflow action can fail after its HTTP trigger has already returned 2xx and an SMTP server can later quarantine or reject delivery。

Required tests add exact Teams envelope/content assertions, verify the generic six-field payload by explicitly selecting `GenericJson`, verify the default and persisted format in `LocalAppSettingsStore`, and retain offline fake HTTP/SMTP boundaries。

## 17. 2026-07 Completed event classification

`NotificationRequestFactory` treats `result.Success` as pipeline completion. A completed result with non-empty `Failures` maps to the completed event for its trigger, uses `Warning` severity, uses message text `Result: Completed`, and retains `FailedAssets`, total failure count and `FirstFailure` details. Only `result.Success = false` maps to a failed event and `Result: Failed`.

Required regression coverage:

- scheduled completed result with record failures → `ScheduledSyncCompleted`, `Warning`, completed subject/message and retained failure details;
- fatal/incomplete result with some successful resource counters → `*SyncFailed`;
- clean completed result → `Information`;
- completed result with warnings → `Warning`.

## 18. 2026-07 Deleted count projection and webhook contract

Change `AteraSnipeSync.Core.Notifications.NotificationRequest` by adding:

```csharp
public required int Deleted { get; init; }
```

`Deleted` is a prepared, non-negative notification metric. `NotificationRequestFactory.CreateForSyncResult` sets it to `result.ImportResult?.DeletedAssets ?? 0`. `CompositeNotificationPublisher.TestAsync` sets it to 0. Every direct test fixture must initialize it explicitly.

`NotificationRequestFactory.BuildMessage` adds `DeletedAssets: {deletedAssets}` after `UpdatedAssets` and before `SkippedAssets`. It must use the same `SnipeImportResult.DeletedAssets` value used by `NotificationRequest.Deleted`; it must not derive the count from actions, failures, status history or message text.

`WebhookNotificationSender.CreateGenericPayload` changes the generic contract to:

```json
{
  "eventType": "...",
  "severity": "...",
  "subject": "...",
  "message": "...",
  "deleted": 3,
  "occurredAtUtc": "..."
}
```

`deleted` is a JSON number and comes directly from `request.Deleted`. The Teams Adaptive Card envelope must not receive an unsupported root property; its existing message TextBlock exposes `DeletedAssets` from the prepared message. No notification path may include delete response bodies, credentials, endpoint URLs or raw asset payloads.

Required offline regressions:

- factory result with `DeletedAssets = 3` produces `request.Deleted == 3` and message line `DeletedAssets: 3`;
- generic fake-handler request contains numeric top-level `deleted: 3` and preserves all existing safe fields;
- test notification generic JSON contains numeric `deleted: 0` through the explicitly initialized request;
- no test contacts real SMTP, webhook, Atera or Snipe-IT endpoints.
