# Notification - 技术规格

## 1. Scope

Notification Module 为 sync runtime 提供通知请求构造、事件过滤和可替换 publisher 边界。

本技术规格基于：

- `docs/module-plans/06-Notification-功能职责.md`
- `docs/AteraSnipeSync_AI_Agent_Master_Plan.md` 的 `Module 6 - Notification Module`
- existing `src/AteraSnipeSync.Core/Notifications/NotificationRequest.cs`
- existing `src/AteraSnipeSync.Core/Notifications/Interfaces/INotificationPublisher.cs`
- existing `src/AteraSnipeSync.Core/Configuration/NotificationConfig.cs`

第一版实现只提供 stub-safe notification pipeline，不实现真实 email、Teams、Slack 或 webhook 发送。

本模块不新增或修改 Atera/Snipe-IT API wire contracts，不新增真实 API 测试。

## 3. Public Interface

### 3.1 `INotificationPublisher`

Existing interface:

```csharp
public interface INotificationPublisher
{
    Task PublishAsync(
        NotificationRequest request,
        CancellationToken cancellationToken);
}
```

Required comment:

```csharp
/// <summary>
/// Publishes a prepared notification request through a concrete sender implementation.
/// </summary>
```

Rules:

- publisher implementations must validate `request`
- publisher implementations must honor cancellation
- publisher implementations must not mutate `NotificationRequest` or attached `SyncRunResult`
- `OperationCanceledException` must propagate
- non-cancellation send failures may throw to the caller after logging

## 4. Public Models

### 4.1 `NotificationRequest`

Existing model:

```csharp
public sealed class NotificationRequest
{
    public required string EventType { get; init; }
    public required string Severity { get; init; }
    public required string Subject { get; init; }
    public required string Message { get; init; }
    public required SyncRunResult? SyncResult { get; init; }
}
```

Required comment:

```csharp
/// <summary>
/// Carries one prepared notification event and optional sync result context to a publisher.
/// </summary>
```

Property rules:

- `EventType` must be a non-blank stable event string from `NotificationEventTypes` when generated by this module
- `Severity` must be `Information`, `Warning`, `Error`, or `Critical`
- `Subject` must be non-blank and short enough for email/webhook titles
- `Message` must be non-blank and safe to log
- `SyncResult` may be `null` for future non-sync notifications

### 4.2 `NotificationConfig`

Existing model:

```csharp
public sealed class NotificationConfig
{
    public required bool Enabled { get; init; }
    public required IReadOnlyList<string> OnEvents { get; init; }
    public string? EmailTo { get; init; }
    public string? WebhookUrl { get; init; }
}
```

Required comment:

```csharp
/// <summary>
/// Configures whether notification events are enabled and which event names may be published.
/// </summary>
```

Rules:

- `Enabled = false` suppresses all notification publishing
- `OnEvents` must not be `null`
- empty `OnEvents` means no events are published
- event matching is trim + ordinal ignore-case
- `EmailTo` and `WebhookUrl` are reserved for future concrete senders
- `NullNotificationPublisher` must not require `EmailTo` or `WebhookUrl`

## 6. Request Factory

### 6.1 `NotificationRequestFactory`

File:

```text
src/AteraSnipeSync.Core/Notifications/NotificationRequestFactory.cs
```

Signature:

```csharp
public static class NotificationRequestFactory
{
    public static NotificationRequest CreateForSyncResult(
        SyncRunResult result,
        string triggeredBy);
}
```

Required comment:

```csharp
/// <summary>
/// Builds safe, user-readable notification requests from completed sync run results.
/// </summary>
```

Validation:

- throw `ArgumentNullException` when `result` is `null`
- throw `ArgumentException` when `triggeredBy` is null/empty/whitespace

Event mapping:

```text
triggeredBy = scheduled, result.Success = true  -> ScheduledSyncCompleted
triggeredBy = scheduled, result.Success = false -> ScheduledSyncFailed
triggeredBy = manual, result.Success = true     -> ManualSyncCompleted
triggeredBy = manual, result.Success = false    -> ManualSyncFailed
triggeredBy = manual-preview, success           -> ManualPreviewCompleted
triggeredBy = manual-preview, failed            -> ManualPreviewFailed
unknown trigger, success                        -> SyncCompleted
unknown trigger, failed                         -> SyncFailed
```

`triggeredBy` comparison must trim and use ordinal ignore-case.

Severity mapping:

- `Information` when `result.Success = true` and `result.Warnings.Count = 0`
- `Warning` when `result.Success = true` and `result.Warnings.Count > 0`
- `Error` when `result.Success = false`
- `Critical` when `result.Success = false` and any `SyncFailure.Code` clearly identifies authentication, authorization, or credential failure

First implementation may treat these failure codes as critical:

- `AuthenticationFailed`
- `Unauthorized`
- `Forbidden`
- `InvalidApiKey`
- `InvalidToken`

Subject mapping:

- scheduled success: `Scheduled sync completed`
- scheduled failure: `Scheduled sync failed`
- manual success: `Manual sync completed`
- manual failure: `Manual sync failed`
- manual preview success: `Manual preview completed`
- manual preview failure: `Manual preview failed`
- generic success: `Sync completed`
- generic failure: `Sync failed`

Message must include:

- event result text
- `StartedAt` in UTC round-trip format
- `FinishedAt` in UTC round-trip format
- pulled count: `result.PullResult?.Summary.AgentCount ?? 0`
- mapped count: `result.ImportBatch?.Summary.MappedAssetCount ?? 0`
- created assets: `result.ImportResult?.CreatedAssets ?? 0`
- updated assets: `result.ImportResult?.UpdatedAssets ?? 0`
- skipped assets: `result.ImportResult?.SkippedAssets ?? 0`
- failed assets: `result.ImportResult?.FailedAssets ?? result.Failures.Count`
- warning count: `result.Warnings.Count`
- first failure stage/code/message when failures exist

Message must not include:

- API keys
- tokens
- authorization headers
- raw Atera JSON
- raw Snipe-IT JSON
- full asset dumps
- full local configuration JSON

## 7. Event Filter

### 7.1 `NotificationEventFilter`

File:

```text
src/AteraSnipeSync.Core/Notifications/NotificationEventFilter.cs
```

Signature:

```csharp
public sealed class NotificationEventFilter
{
    public bool ShouldPublish(
        NotificationConfig config,
        NotificationRequest request);
}
```

Required comment:

```csharp
/// <summary>
/// Applies notification configuration to decide whether a prepared event should be published.
/// </summary>
```

Validation:

- throw `ArgumentNullException` when `config` is `null`
- throw `ArgumentNullException` when `request` is `null`
- throw `ArgumentException` when `config.OnEvents` is `null`
- throw `ArgumentException` when `request.EventType` is blank

Rules:

- return `false` when `config.Enabled = false`
- return `false` when `config.OnEvents.Count = 0`
- return `true` only when any configured event equals `request.EventType` after trim and ordinal ignore-case comparison
- ignore blank event names in `OnEvents`

This class must not call any external service and must not log secrets.

## 8. Null Publisher

### 8.1 `NullNotificationPublisher`

File:

```text
src/AteraSnipeSync.Core/Notifications/NullNotificationPublisher.cs
```

Signature:

```csharp
public sealed class NullNotificationPublisher : INotificationPublisher
{
    public NullNotificationPublisher(
        ILogger<NullNotificationPublisher> logger);

    public Task PublishAsync(
        NotificationRequest request,
        CancellationToken cancellationToken);
}
```

Required comment:

```csharp
/// <summary>
/// Accepts notification requests without sending external messages, providing a safe default publisher.
/// </summary>
```

Constructor responsibilities:

- validate `logger` is not null

`PublishAsync` responsibilities:

- validate `request` is not null
- validate `EventType`, `Severity`, `Subject`, and `Message` are non-blank
- call `cancellationToken.ThrowIfCancellationRequested()` before logging
- write a debug log with event type and severity
- do not call HTTP, SMTP, file system, Atera API, Snipe-IT API, Teams, Slack, or any webhook
- return `Task.CompletedTask`

Logs must not include raw `Message` if future messages may contain user-entered text. First implementation may log only event type, severity, and subject.

## 9. Integration Rules

Worker Scheduler and future manual sync flows may use:

```text
SyncRunResult
  -> NotificationRequestFactory.CreateForSyncResult(...)
  -> NotificationEventFilter.ShouldPublish(...)
  -> INotificationPublisher.PublishAsync(...)
```

Notification publish should happen after status save when a completed run result exists.

If notification publishing fails:

- log the notification failure
- do not rewrite the original `SyncRunResult`
- let the caller decide whether the runtime loop continues

Cancellation should stop publishing immediately and propagate `OperationCanceledException`.

## 10. Logging

Use `ILogger<NullNotificationPublisher>` for null publisher logs.

Required log events:

- Debug: null publisher accepted a notification request
- Error: future concrete senders should log failed send attempts before throwing

Logs must not include secrets, auth headers, raw payloads, full config content, or full asset inventories.

## 12. Acceptance Criteria

Implementation is accepted when:

- `dotnet build` succeeds
- `dotnet test` succeeds
- all required Notification tests pass
- generated event types match `NotificationEventTypes`
- severity mapping follows this spec
- disabled or unmatched config suppresses publishing
- `NullNotificationPublisher` sends no external traffic
- cancellation propagates as `OperationCanceledException`
- notification messages do not persist or log secrets/raw payloads
- `progress.md` is updated in the same work session as any production code or automated test changes

## 14. 2026-07 Scheduler 通知集成

`SyncScheduler` constructor 新增 `NotificationConfig notificationConfig` 与 `NotificationEventFilter notificationEventFilter`。单次 run 后执行：

```csharp
var request = NotificationRequestFactory.CreateForSyncResult(result, "scheduled");
if (_notificationEventFilter.ShouldPublish(_notificationConfig, request))
    await _notificationPublisher.PublishAsync(request, cancellationToken);
```

`NotificationRequestFactory` 的 critical code 判定使用 `code.Equals(name)` 或 `code.EndsWith("." + name)`；不得 substring 匹配。publisher 异常由 scheduler 记录，当前 run 返回 false，但后台 loop 继续。
