Module 4: Sync Orchestrator 流程复习

本页把模块 4 需要完成的事情画成流程图：它不直接访问 Atera 或 Snipe-IT API，而是把 Atera Pull、Reconstruction、Snipe Import 三个核心模块串成一次完整 sync run，并输出结构化结果。

1. 一句话定位

模块边界

只负责编排

接收 SyncRunRequest，按固定顺序调用三个核心模块，不理解 API 细节、不做字段 mapping、不做 UI。

核心产物

生成 SyncRunResult

记录 started/finished 时间，保存 pull、batch、import result，并聚合 warnings 和 failures。

安全开关

传递 Dry Run

SyncRunOptions.DryRun 要影响 import 行为，manual preview、scheduled sync、test run 都走同一个入口。

2. 主流程图

1收到请求

入口是 ISyncOrchestrator.RunOnceAsync(request, ct)。

  • 确认 request 不为空
  • 读取 TriggeredBy
  • 记录 StartedAt

2Atera Pull

调用 IAteraClient.PullInventoryAsync 拉取完整 Atera inventory。

  • 成功时拿到 AteraPullResult
  • 收集 pull warnings
  • 失败时立刻停止后续步骤

3Reconstruction

调用 IInventoryMapper.Map 把 pull result 转成 SnipeImportBatch。

  • 不调用外部系统
  • 收集 mapping warnings
  • map 失败时不进入 import

4Snipe Import

调用 ISnipeImporter.ImportAsync，把 batch 交给 Snipe Import 模块处理。

  • 确保 dry-run 意图传入 options
  • 收集 import warnings
  • 把 import failures 转成 run failures

5汇总结果

组合三个阶段的输出，生成完整 SyncRunResult。

  • Success 由 failures 决定
  • 保留可用的中间结果
  • 记录 FinishedAt

6交给调用方

返回结果给 Worker、TrayApp、测试入口或未来 Status Store。

  • Orchestrator 不保存文件
  • 不发 email
  • 不直接更新 UI

重点记忆：模块 4 是 pipeline 控制器。它决定“下一步能不能继续”，但不决定 Atera API 字段、Snipe-IT payload 或 UI 展示。

3. 失败分支

1Pull 失败

例如认证失败、网络失败、分页无法确认或 response 无法解析。

  • 不调用 mapper
  • 不调用 importer
  • PullResult 通常为空
  • 新增 SyncFailure.Stage = "AteraPull"

2Map 失败

例如 mapper 抛出不可恢复异常，或未来技术规格定义 batch 无效。

  • 保留已成功的 PullResult
  • 不调用 importer
  • ImportBatch 为空或不可用
  • 新增 SyncFailure.Stage = "Reconstruction"

3Import 失败

包括 importer 抛异常，或返回的 SnipeImportResult.Failures 有失败资产。

  • 保留 pull result 和 import batch
  • 返回 import result 中的 action/failure 信息
  • Success = false
  • 新增 SyncFailure.Stage = "SnipeImport"

  • 现在这两个文件只有标题，按项目规则，正式写生产代码前必须先补齐。
代码阶段

实现 SyncOrchestrator

  • 新增 SyncOrchestrator 类，实现 ISyncOrchestrator。
  • 构造函数注入 IAteraClient、IInventoryMapper、ISnipeImporter、可测试时间来源、logger。
  • 在 RunOnceAsync 内按 pull -> map -> import 顺序执行。
  • 捕获阶段异常并转换为 SyncFailure，不要吞掉 cancellation。
结果阶段

统一聚合输出

  • 把 AteraPullResult.Warnings、SnipeImportBatch.Warnings、SnipeImportResult.Warnings 合并。
  • 把 importer 的 asset-level failures 映射到 run-level failures，保留原始 import result。
  • 无失败时 Success = true；有任一阶段失败时 Success = false。
  • 始终填充 StartedAt 和 FinishedAt。
测试阶段

写 orchestrator 单元测试

  • 正常流程按顺序调用 pull、mapper、importer。
  • pull 失败时 mapper/importer 都不调用。
  • map 失败时 importer 不调用。
  • import 失败时返回 structured failure。
  • dry-run 与 triggeredBy 正确传递或保留。

5. 合约速查

对象 模块 4 怎么用 注意点
SyncRunRequest 一次 sync run 的完整输入，包含 Atera、Mapping、SnipeIt、Sync 四组配置。 Orchestrator 不从全局配置读取 secret 或设置项。
SyncRunOptions 携带 DryRun 和 TriggeredBy。 TriggeredBy 可为 scheduled、manual、test；用于结果追踪和日志。
AteraPullResult pull 成功后的原始 inventory 快照。 失败时不应把半成品继续传给 mapper。
SnipeImportBatch Reconstruction 输出，Snipe Import 输入。 Orchestrator 不修改字段 mapping 规则。
SnipeImportResult import 模块返回 create/update/skip/failure/action/warning 统计。 即使 import 有失败，也要保留 result 供 UI/status/report 使用。
SyncRunResult 模块 4 的最终返回值。 后续 Status Store、Scheduler、TrayApp 都应消费这个结构化结果。

6. 责任边界

应该做

Orchestrator 内部职责

  • 控制一次 run 的生命周期。
  • 按顺序调用三个核心模块。
  • 聚合 warnings 和 failures。
  • 生成结构化结果。
  • 保留 dry-run 和 triggeredBy 语义。
不应该做

必须留给其他模块

  • 不直接写 Atera API 或 Snipe-IT API。
  • 不做 Atera 到 Snipe-IT 的字段映射。
  • 不保存 status 文件。
  • 不发送 email 或通知。
  • 不包含 TrayApp UI 逻辑。
